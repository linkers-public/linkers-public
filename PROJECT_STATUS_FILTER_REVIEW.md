# 프로젝트 상태 필터 검토 보고서

## 현재 상태 분류

### 필터 기준: `counsel.counsel_status`
- **대기중** (`pending`): 프로젝트 생성 직후, 견적 요청 전
- **모집중** (`recruiting`): 특정 팀에게 견적 요청한 후
- **종료** (`end`): 프로젝트 완료

## 데이터 현황

### 실제 데이터 분석
```
pending: 4개 (팀 지정 없음, 견적서 0개)
recruiting: 9개 (팀 지정 없음, 견적서 0개) ⚠️ 이상함
end: 0개
```

### 문제점 발견

1. **데이터 불일치**
   - `recruiting` 상태인데 `requested_team_id`가 NULL인 프로젝트가 9개
   - 이론적으로는 `recruiting` = 특정 팀 지정했을 때인데, 실제로는 그렇지 않음

2. **상태 구분이 모호함**
   - `pending`과 `recruiting`의 차이가 사용자 입장에서 명확하지 않음
   - 둘 다 견적서가 0개면 사실상 같은 상태

3. **실용성 부족**
   - "종료" 필터: 현재 0개라서 의미 없음
   - "대기중" vs "모집중": 실제 차이가 없어 보임

## MVP 관점에서의 검토

### 현재 MVP 목표
- **견적서 열람** 기능만 구현
- 계약 진행 등 복잡한 워크플로우는 제외

### 상태 필터의 필요성
1. **필요한 경우**
   - 견적서가 있는 프로젝트 / 없는 프로젝트 구분
   - 진행 중인 프로젝트 / 완료된 프로젝트 구분

2. **불필요한 경우**
   - `pending` vs `recruiting` 구분 (실제 차이 없음)
   - 복잡한 상태 관리 (MVP에서는 과함)

## 개선 제안

### 옵션 1: 필터 단순화 (권장)
```
- 전체
- 견적서 있음 (estimate_count > 0)
- 견적서 없음 (estimate_count = 0)
- 완료 (end)
```

### 옵션 2: 필터 제거
- 상태 필터 제거
- 상태는 뱃지로만 표시
- 검색/정렬 기능만 유지

### 옵션 3: 현재 유지 + 데이터 정리
- 현재 필터 유지
- `recruiting` 상태인데 `requested_team_id`가 NULL인 경우 `pending`으로 수정
- 상태 변경 로직 명확화

## 권장 사항

**MVP에서는 옵션 1 (단순화) 권장**

이유:
1. 사용자가 이해하기 쉬움
2. 실제로 유용한 구분 (견적서 있음/없음)
3. 데이터 불일치 문제 해결
4. 복잡한 상태 관리 불필요
